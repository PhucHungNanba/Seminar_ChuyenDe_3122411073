# 05.4.2 Feature: Rx Upload & Pharmacist Split-screen Approval (Checklist & Test Cases)

Checklist hướng dẫn triển khai nghiệp vụ tải ảnh đơn thuốc của khách hàng lên Cloudinary CDN và thiết kế màn hình kiểm duyệt chia đôi (Split-screen) dành riêng cho Dược sĩ bốc thuốc và tạo báo giá thực tế tại Admin Portal.

---

## 📋 Danh sách các đầu việc thực thi (Checklist)

- [x] **Bước 1: Triển khai API Upload Đơn thuốc phía Backend**
  - Xây dựng route `POST /api/prescriptions` tại `order-service`.
  - Sử dụng middleware `multer` để nhận file tải lên lưu trong Buffer tạm thời.
  - Gọi hàm stream của Cloudinary SDK đẩy nhị phân ảnh trực tiếp lên máy chủ đám mây, nhận về `secure_url`.
  - Khởi tạo bản ghi mới trong bảng `prescriptionvaults` với `thumbnailUrl` là link vừa nhận và `status` mặc định là `PENDING`.

- [x] **Bước 2: Tự động tính toán hạn dùng đơn thuốc y tế**
  - Thiết kế API tiếp nhận các trường text tùy chọn do khách nhập: `doctorName`, `hospital`, `diagnosis`, `issueDate` (Ngày bác sĩ cấp đơn).
  - Viết logic Mongoose Pre-save: Nếu có trường `issueDate`, hệ thống tự động cộng thêm `30 ngày` để lưu trữ vào trường `expiryDate`.

- [x] **Bước 3: Phát triển Giao diện Tải đơn thuốc phía Client**
  - Cập nhật component `QuickPrescriptionModal.tsx` và trang `MedicineDetailPage.tsx` (Mục gửi đơn thuốc Rx).
  - Bổ dung Form nhập liệu các trường văn bản: Bác sĩ điều trị, Bệnh viện cấp đơn, Chẩn đoán bệnh, Ngày cấp đơn (Sử dụng ô chọn ngày `<input type="date" />`).
  - Đóng gói toàn bộ file ảnh và text bằng `FormData` để gửi HTTP Request lên API `/api/prescriptions`.

- [x] **Bước 4: Thiết kế Màn hình Duyệt đơn Chia đôi (Split-screen) cho Dược sĩ**
  - Tại trang `RxApprovalPage.tsx` của Admin Portal, xây dựng giao diện phân chia Responsive làm 2 cột:
    - [x] **Cột bên trái (Soi đơn thuốc):** Hiển thị ảnh chụp đơn thuốc gốc của khách hàng lấy từ URL Cloudinary (`thumbnailUrl`). Bổ sung thanh điều khiển nhỏ cho phép người dùng click Zoom In/Out và Xoay ảnh (Rotate) sử dụng CSS transform để dễ đọc chữ viết tay của bác sĩ.
    - [x] **Cột bên phải (Bốc thuốc tạo báo giá):** Form tìm kiếm nhanh sản phẩm y tế bằng cách gọi API của `Product Service` thông qua API Gateway. Dược sĩ gõ tên hoạt chất, hiển thị danh sách gợi ý thuốc thật đang có trong kho hàng, bấm chọn, điền số lượng bốc và giá bán đề xuất để đưa vào bảng báo giá chi tiết.

- [x] **Bước 5: Ràng buộc đối chiếu Tồn kho thực tế (Inventory Match)**
  - Khi Dược sĩ thêm một loại thuốc vào báo giá, hệ thống so sánh số lượng Dược sĩ bốc với thông tin `inventory.stock_quantity` của chi nhánh.
  - Nếu số lượng bốc lớn hơn số lượng trong kho, hiển thị ngay Badge cảnh báo màu vàng nổi bật: *"⚠️ Vượt quá tồn kho thực tế"*, nhưng vẫn cho phép Dược sĩ chèn vào báo giá (để phục vụ điều chuyển kho sau này).

- [x] **Bước 6: API Duyệt đơn & Kết xuất Báo giá**
  - Thiết kế API `PATCH /api/prescriptions/:id/approve` để Dược sĩ gửi bảng thuốc vừa bốc lên Backend.
  - Backend cập nhật mảng `medicines` bốc thực tế vào DB và chuyển trạng thái đơn sang `APPROVED` (Trạng thái đơn hàng tổng chuyển dịch sang `QUOTED`).

## 🧪 HOẠT ĐỘNG KIỂM THỬ VI MÔ (Micro-Testing Steps)

> [!NOTE]
> ### 🧪 Kiểm thử Tích hợp API Upload (Supertest)
> - **Kịch bản:** Viết file test `tests/rx-upload.test.ts`. Sử dụng `supertest` giả lập upload một tệp tin ảnh nhị phân thực tế đính kèm các trường text (với `issueDate` thiết lập là ngày hiện tại).
> - **Tiêu chí Pass:**
>   - API phản hồi trạng thái `HTTP 201 Created` thành công.
>   - Record trong database được khởi tạo chứa link ảnh thật từ Cloudinary.
>   - Trường `expiryDate` tự động nhảy thêm chính xác 30 ngày so với `issueDate` đã gửi.
>
> ### 🧪 Kiểm thử Giao diện (Browser Subagent)
> - **Kịch bản:** Kích hoạt `Browser Subagent` mở cổng `5174`, truy cập `/admin/rx-approval`. Click chọn xem chi tiết một đơn thuốc đang chờ duyệt.
> - **Tiêu chí Pass:**
>   - Giao diện Split-screen được chia làm 2 cột cân đối.
>   - Các nút xoay ảnh và zoom ảnh bên trái hoạt động mượt mà không vỡ bố cục.
>   - Ô tìm kiếm thuốc bên phải tự động hiển thị danh sách thuốc mồi khi gõ từ khóa, hiển thị đúng cảnh báo tồn kho nếu vượt quá `stock_quantity` có trong DB.
